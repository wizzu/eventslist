- [x] **Search render freeze — yield trick**: When the debounced query fires, the main thread immediately does all filtering and DOM updates synchronously, blocking user input and freezing the UI for roughly 1 second (measured at ~2000 events). The user may still be typing (debounce fired early on slow input) and the app becomes unresponsive mid-keystroke. The goal is to keep the UI visually alive during the update — showing a loading/thinking state rather than a hard freeze — even if the total time stays the same. The yield trick: instead of setting `this.query = val` directly in the debounce callback, first update a `searching` flag so Alpine renders a loading indicator, then use `setTimeout(0)` (or `requestAnimationFrame`) to defer the actual query assignment to the next event loop tick. This gives the browser a chance to paint the loading state before the heavy render work begins. Note: this is a UX improvement, not a performance fix — the work takes the same time, it just feels less broken. Should be implemented before investigating more structural solutions. See also: virtual scrolling TODO below. **Tried and reverted:** implemented with `requestAnimationFrame` wrapping the query assignment; had no practical effect. CSS animations are also main-thread-driven, so the `search-pending` animation cannot play during the freeze — the browser gets one frame before the 1s render blocks everything, and by the time it unblocks `searching` is already `false`. A yield only helps if the work is broken into small interleaved chunks; a single 1s lump is still a 1s freeze regardless.

- [x] **Search render freeze — investigate virtual scrolling** ⚠️ INVESTIGATED — DECIDED NOT TO IMPLEMENT: The root cause of the ~1s UI freeze when search changes is DOM size. Alpine's `x-for` renders all matching events as real DOM nodes at once — at 2000 events (growing ~100/year toward ~5000), each with multiple elements (date, performers, venue, badges), that's tens of thousands of DOM nodes being created or diffed on every query change. JS filtering itself is fast (milliseconds); the bottleneck is the browser's layout and paint pass over the full rendered list. Virtual scrolling solves this by only rendering the ~20–30 rows visible in the viewport at any time, keeping the DOM constant-size regardless of result count. **Investigation outcome:** virtual scrolling was assessed as not viable for this app in its current form. The trade-offs in combination are too costly: (1) **Ctrl+F (browser find-in-page) breaks** — only the rendered DOM slice is searchable; rows outside the viewport are invisible to it. Ctrl+F is an occasional but legitimate use case. (2) **Dual scroll mode complexity** — desktop uses `.left-col` scroll, mobile uses `window` scroll; virtual scroll logic must hook into both. (3) **Variable row height** — rows with multiple performers or event name prefixes are taller than single-performer rows; padding-based spacers use an estimated average height, causing scrollbar drift and positional inaccuracy as actual heights diverge. (4) **No existing library** — no Alpine-compatible virtual scroll library exists; implementation would be fully custom (~30–40 lines of JS + template changes). Each trade-off might be tolerable alone; together they make the cure worse than the disease. The freeze is real but contained: it mainly hits the full listing and clearing search; typical usage (searching for a performer or venue) returns a small result set and is fast. An `innerHTML`-based rendering approach was also considered (bypass Alpine diffing by computing the list as an HTML string) but was assessed as likely ineffective since the bottleneck is probably browser layout/paint, not Alpine's diffing overhead. Revisit if the dataset grows significantly beyond 5000 events.
